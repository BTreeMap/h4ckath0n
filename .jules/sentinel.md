## 2024-05-07 - Fix timing attack vulnerability in password authentication
**Vulnerability:** The `authenticate_user` function immediately returned `None` when a user was not found or when a user lacked a password hash. This allowed an attacker to enumerate valid email addresses based on response times, because verifying an Argon2 hash takes significantly longer than checking if a user exists.
**Learning:** Argon2id hashing is intentionally slow to protect against brute-force attacks. When we fail early before verifying a password, we leak information. Timing attacks require mitigations even if standard conditionals are present. Furthermore, using a structurally invalid dummy hash with argon2id raises an exception rather than doing a timing-safe compare, so we must use a real hash to mitigate properly.
**Prevention:** Always verify a password hash (either the actual hash or a structurally valid dummy hash) regardless of whether the user exists or not. This ensures the execution time is consistent, hiding the existence of the account.
